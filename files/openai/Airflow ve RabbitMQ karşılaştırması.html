Apache Airflow ile RabbitMQ farklı amaçlara hizmet eden sistemlerdir. Hangisinin "daha iyi" olduğu sorusu, ne yapmak istediğine bağlıdır. Kısaca açıklayalım:

🌀 Apache Airflow Nedir?

  • Ne işe yarar?
    Zamanlanabilir iş akışları (workflows) oluşturmak ve bu işleri yönetmek için kullanılır.

  • Kullanım alanları:

    • ETL/ELT işlemleri

    • Veri boru hatları (data pipelines)

    • Makine öğrenimi model eğitimlerinin otomasyonu

    • Zamanlanmış raporlar

  • Özellikleri:

    • DAG (Directed Acyclic Graph) temelli yapı

    • Gantt chart'larla görselleştirme

    • Task retry, SLA, bağımlılık yönetimi

    • Executor'lar sayesinde dağıtık çalışabilir (örn. Celery, Kubernetes)

🐇 RabbitMQ Nedir?

  • Ne işe yarar?
    Mesaj kuyruğu sistemidir (message broker). Sistemler arasında mesajlaşmayı sağlar.

  • Kullanım alanları:

    • Mikroservis mimarilerinde servisler arası iletişim

    • Asenkron görev sıralama (örn. e-posta gönderme, video işleme)

    • Olay tabanlı sistemler

  • Özellikleri:

    • AMQP protokolü

    • Dayanıklılık (durability), ack mekanizması

    • Publish/Subscribe, Queue/Worker gibi desenler

🧩 Not: Birlikte Kullanım da Mümkün

İki sistem birlikte de kullanılabilir:

  • Airflow, Celery Executor kullanırken RabbitMQ (veya Redis) mesajlaşma altyapısı olarak kullanılabilir.

  • Bir DAG içinde, bir task RabbitMQ'ya mesaj gönderip başka bir sistemi tetikleyebilir.

Sonuç

🔸 Eğer zamanlanabilir veri işleri, iş akışları veya ETL süreçleri ile ilgileniyorsan:
👉 Airflow

🔸 Eğer asenkron mesajlaşma, mikroservisler arası iletişim ya da event-driven mimari kuruyorsan:
👉 RabbitMQ

İhtiyacını söylersen, hangi aracı tercih etmen gerektiğini daha net söyleyebilirim.

🎯 Neden RabbitMQ Mantıklı?

Çünkü senin senaryonda:

  • Yeni dosya geldiğinde bir "olay" oluşuyor → Event

  • Bu olay, birden fazla mikroservise "bildirilmek" isteniyor → Pub/Sub

  • Mikroservislerin birbirinden bağımsız olması isteniyor → Loosely Coupled Architecture

RabbitMQ, tam da bu tarz mesajlaşma ve olay yayını (publish/subscribe) için tasarlanmıştır.

🧱 Mimaride Ne Olur?

1. Watcher Service (Dosya İzleyici)

  • Belirli bir adresi izler (örn. klasör, S3 bucket, FTP, HTTP endpoint).

  • Yeni bir dosya geldiğinde bir mesaj oluşturur ve RabbitMQ'ya yayınlar.

json
{
  "event": "file_uploaded",
  "filename": "invoice_2024_09_01.csv",
  "path": "/uploads/invoices/",
  "timestamp": "2025-09-01T12:30:00Z"
}

2. RabbitMQ Broker

  • Bu mesajı kuyruklara yönlendirir.

  • İlgili mikroservisler (subscribers) bu mesajı alır.

3. İlgili Mikroservisler

  • Dosyayla ilgili işlem yapacak servisler mesajı alır:

    • FileProcessorService: Dosyayı okur, işler.

    • NotificationService: Kullanıcıya e-posta gönderir.

    • AuditService: Log kaydı oluşturur.

📌 Airflow Gerekli mi?

Hayır, bu senaryo için değil.

Airflow, dosya geldiğinde iş başlatmak için kullanılabilir, ama daha çok:

  • Zamanlanmış işler,

  • ETL boru hatları,

  • Bağımlılık yönetimi,

  • Uzun süreli batch işler

için uygundur.

Senin senaryon "event-driven" (dosya gelince hemen reaksiyon) olduğu için Airflow fazla karmaşık ve gecikmeli olur.

🧠 Alternatifler:

  • RabbitMQ → Eğer anında, garantili mesaj teslimi istiyorsan (ack vs.)

  • Apache Kafka → Eğer yüksek hacimli veriyle uğraşıyorsan ve mesajların sırası önemliyse

  • Amazon SQS + Lambda → AWS üzerinde çalışıyorsan bu mimari de iyi bir alternatiftir.

  • Redis Streams → Daha hafif yapılı event queue ihtiyacın varsa

⚙️ Örnek Akış Açıklaması

  1. File Watcher servisi belirli bir dizini izler. (örn: inotify, watchdog, ya da S3 event notification)

  2. Yeni bir dosya geldiğinde bir file_uploaded olayı oluşturur.

  3. Bu olay, RabbitMQ'ya bir mesaj olarak yayınlanır (publish).

  4. RabbitMQ bu mesajı ilgili kuyruklara gönderir (fanout, topic, ya da direct exchange kullanılır).

  5. Abone olan mikroservisler bu mesajı alır ve işlem yapar.

🧪 Örnek: Basit Kod İskeleti

🔍 1. File Watcher (Python ile)

python
# file_watcher.py
import pika
import os
from watchdog.observers import Observer
from watchdog.events import FileSystemEventHandler
import json

RABBITMQ_HOST = "localhost"
QUEUE_NAME = "file_events"

class FileHandler(FileSystemEventHandler):
    def on_created(self, event):
        if event.is_directory:
            return
        
        message = {
            "event": "file_uploaded",
            "filename": os.path.basename(event.src_path),
            "path": event.src_path
        }

        connection = pika.BlockingConnection(pika.ConnectionParameters(RABBITMQ_HOST))
        channel = connection.channel()
        channel.queue_declare(queue=QUEUE_NAME, durable=True)
        channel.basic_publish(
            exchange='',
            routing_key=QUEUE_NAME,
            body=json.dumps(message)
        )
        connection.close()
        print(f"Sent message: {message}")

if __name__ == "__main__":
    path_to_watch = "/uploads/"
    observer = Observer()
    observer.schedule(FileHandler(), path=path_to_watch, recursive=False)
    observer.start()
    print(f"Watching {path_to_watch} for new files...")
    observer.join()

📥 2. Bir Mikroservisin Dinlemesi (örn: File Processor)

python
# file_processor.py
import pika
import json

RABBITMQ_HOST = "localhost"
QUEUE_NAME = "file_events"

def callback(ch, method, properties, body):
    message = json.loads(body)
    print(f"Processing file: {message['filename']} at {message['path']}")
    # Dosya işleme logic'i buraya
    ch.basic_ack(delivery_tag=method.delivery_tag)

connection = pika.BlockingConnection(pika.ConnectionParameters(RABBITMQ_HOST))
channel = connection.channel()
channel.queue_declare(queue=QUEUE_NAME, durable=True)
channel.basic_qos(prefetch_count=1)
channel.basic_consume(queue=QUEUE_NAME, on_message_callback=callback)

print("File Processor listening for events...")
channel.start_consuming()

💡 RabbitMQ Exchange Yapılandırması (Opsiyonel)

Eğer birden fazla mikroservis dinleyecekse (fanout gibi), exchange tanımlaman gerekebilir:

python
channel.exchange_declare(exchange='file_exchange', exchange_type='fanout')
channel.queue_bind(exchange='file_exchange', queue='file_processor_queue')

📝 Docker-Compose Yapılandırması

Aşağıdaki docker-compose.yml dosyasını kullanarak, RabbitMQ’yu ve Python servislerini aynı ağ üzerinde çalıştırabilirsin. Bu dosya, RabbitMQ'nun yanı sıra, Python mikroservislerinin her birini konteyner olarak başlatacak.

yaml
version: '3.8'

services:
  rabbitmq:
    image: rabbitmq:3-management
    container_name: rabbitmq
    ports:
      - "5672:5672"      # RabbitMQ mesaj kuyruğu portu
      - "15672:15672"    # RabbitMQ web yönetim arayüzü
    volumes:
      - rabbitmq_data:/var/lib/rabbitmq  # Veriyi kalıcı tutmak için volume
    environment:
      - RABBITMQ_DEFAULT_USER=guest
      - RABBITMQ_DEFAULT_PASS=guest
    networks:
      - app_network

  file_watcher:
    build:
      context: ./file_watcher  # File watcher için bulunduğun klasör
    container_name: file_watcher
    volumes:
      - ./uploads:/uploads     # Kendi dosya dizinini bağla
    depends_on:
      - rabbitmq
    networks:
      - app_network

  file_processor:
    build:
      context: ./file_processor  # File processor için bulunduğun klasör
    container_name: file_processor
    depends_on:
      - rabbitmq
    networks:
      - app_network

volumes:
  rabbitmq_data:
    driver: local

networks:
  app_network:
    driver: bridge

📌 Offline Bağımlılıkları Nasıl Yüklerim?

Eğer internet bağlantısı yoksa ve Python bağımlılıklarını önceden indirdiysen, requirements.txt içindeki paketleri şu şekilde yükleyebilirsin:

  1. Gereksinimlerinizi yerel olarak indirdikten sonra, bu dosyaları bir klasöre yerleştir.

  2. Dockerfile'da bu dosyaları kopyalayarak aşağıdaki gibi yükleyebilirsin:

Dockerfile
COPY ./local_packages /app/packages
RUN pip install --no-index --find-links=/app/packages -r requirements.txt

Böylece Python bağımlılıklarını offline bir şekilde yükleyebilirsin.
